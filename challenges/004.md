# Challenge 4: Rate Limiter Middleware

## Business Requirement
Protect an API from abuse by limiting how many requests a single user can make within a time window.

## Components to Build

The system needs three parts:
1. A way to mark which endpoints need protection
2. A mechanism to track and enforce limits
3. Test endpoints to verify protection works

## Business Requirements

### Requirement 1: Identify protected endpoints
Business stakeholders must be able to mark specific API endpoints as rate-limited without changing the endpoint's core logic.

### Requirement 2: Default reasonable limits
By default, any protected endpoint should allow no more than 5 requests per minute from a single user.

### Requirement 3: Configurable limits
Different endpoints may need different limits (e.g., login endpoints might need stricter limits than search endpoints).

### Requirement 4: Track users by IP address
The system must identify each user by their IP address to apply limits fairly.

### Requirement 5: Use rolling time windows
The limit applies to any 60-second window, not fixed calendar minutes. This prevents users from making 5 requests at 0:59 and another 5 at 1:01.

### Requirement 6: Reject excessive requests
When a user exceeds the limit, the system must:
- Return HTTP 429 (Too Many Requests)
- Tell the user how many seconds to wait before trying again
- Include a Retry-After header for automated clients

### Requirement 7: Provide clear limit information
The rejection message must include:
- That the rate limit was exceeded
- How many seconds until they can try again
- What the limit is
- What the time window is

### Requirement 8: Never limit unmarked endpoints
Some endpoints (like health checks or static assets) must never be rate-limited.

### Requirement 9: Clean up old tracking data (optional)
The system should periodically clean up tracking data for inactive users to prevent memory issues.

## Acceptance Tests

# Test 1: Unprotected endpoint handles unlimited traffic
GET /unlimited (send 100 rapid requests)
Expected: All requests succeed

# Test 2: Protected endpoint allows first 5 requests
GET /limited (send 5 requests within 1 second)
Expected: All 5 succeed

# Test 3: 6th request within same minute is blocked
GET /limited (send 6th request within 60 seconds of the first)
Expected: System returns 429 with retry information

# Test 4: Limit resets after time window passes
Wait 60 seconds
GET /limited
Expected: Request succeeds (new window)

# Test 5: Different users have independent limits
User A from IP 192.168.1.1 sends 5 requests
User B from IP 192.168.1.2 sends 5 requests
Expected: Both users can make 5 requests each

# Test 6: Rolling window behavior
# Send requests at second 0, 10, 20, 30, 40
# Send 6th request at second 50
Expected: Request at second 50 is rejected (still 5 requests in last 60 seconds)
# Send request at second 61
Expected: Request succeeds (request at second 0 is now outside window)

# Test 7: Different limits for different endpoints
# Endpoint A: 2 requests per 10 seconds
# Endpoint B: 10 requests per 60 seconds

Expected: Each endpoint enforces its own limit independently
