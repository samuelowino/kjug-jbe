# Challenge 2: Task Manager with Validation

## Business Requirement
Provide a task management system for office workers to create, track, and complete their daily tasks.

## Endpoints

| Method    | Endpoint                  | Business Purpose |
|-----------|---------------------------|------------------|
| POST      | `/tasks`                  | Create a new work task |
| GET       | `/tasks`                  | View all tasks |
| GET       | `/tasks/pending`          | View only unfinished tasks |
| PUT       | `/tasks/{id}/complete`    | Mark a task as done |

## Data Model Business Rules

- Each task must have a unique identifier
- Each task must have a title describing the work
- Each task can have a priority (Low, Medium, High)
- Each task can have a due date
- Each task tracks whether it is complete

## Business Requirements

### Requirement 1: Create valid tasks only
The system must only accept tasks with:
- A title that is not empty and not excessively long (max 100 characters)
- A valid priority level (Low, Medium, or High only)
- A due date that is today or in the future (no past due dates allowed at creation)

### Requirement 2: Auto-assign task ID
The system must automatically generate a unique ID for each new task.

### Requirement 3: Track all created tasks
The system must store all tasks and allow users to retrieve the full list at any time.

### Requirement 4: Filter incomplete work
The system must provide a view showing only tasks that have not been completed yet, helping users focus on outstanding work.

### Requirement 5: Mark completion safely
The system must allow users to mark tasks as complete, but must:
- Reject attempts to complete non-existent tasks
- Reject attempts to complete already-completed tasks (prevent duplicate actions)

### Requirement 6: Provide helpful error messages
When validation fails, the system must tell the user exactly which field caused the problem and why.

## Acceptance Tests

# Test 1: Manager creates a valid task
POST /tasks
Body: {"title":"Review Q2 budget","priority":"HIGH","dueDate":"2026-04-25"}
Expected: System creates task and returns it with a generated ID

# Test 2: User tries to create task with empty title
POST /tasks
Body: {"title":"","priority":"LOW","dueDate":"2026-04-25"}
Expected: System rejects with error about title being required

# Test 3: User tries to set a past due date
POST /tasks
Body: {"title":"Old meeting notes","priority":"MEDIUM","dueDate":"2020-01-01"}
Expected: System rejects because due date must be today or future

# Test 4: User tries invalid priority
POST /tasks
Body: {"title":"Important work","priority":"CRITICAL","dueDate":"2026-04-25"}
Expected: System rejects with list of allowed priorities

# Test 5: User views all tasks
GET /tasks
Expected: System returns all tasks in the system

# Test 6: User views only pending tasks
GET /tasks/pending
Expected: System returns only tasks where completed = false

# Test 7: User completes a task
PUT /tasks/1/complete
Expected: System marks task as complete and returns updated task

# Test 8: User tries to complete already-completed task
PUT /tasks/1/complete
Expected: System rejects with conflict error

# Test 9: User tries to complete non-existent task
PUT /tasks/999/complete
Expected: System rejects with not found error
